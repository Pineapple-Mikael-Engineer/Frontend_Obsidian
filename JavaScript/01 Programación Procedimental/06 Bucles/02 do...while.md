# do...while

> [!definicion]
> `do...while` ejecuta el cuerpo **al menos una vez** y luego evalúa la condición. Si la condición es verdadera, repite el cuerpo; si es falsa, termina. Sintaxis: `do { cuerpo } while (condición);`. La diferencia crítica con [[01 while]] es que la condición se evalúa **después** de la primera ejecución, no antes.

```js
let i = 0;
do {
  console.log(i); // 0, 1, 2
  i++;
} while (i < 3);

// Condición falsa desde el inicio → cuerpo se ejecuta igualmente una vez
let j = 10;
do {
  console.log("ejecutado una vez:", j); // "ejecutado una vez: 10"
} while (j < 3);
```

## Cómo funciona por dentro

El motor ejecuta el cuerpo sin evaluar la condición. Al llegar al `while (condición)`, aplica ToBoolean al resultado. Si es `true`, vuelve al inicio del `do`; si es `false`, sale. El punto y coma final después de `while (condición)` es **obligatorio** — es parte de la sintaxis.

## Cuándo usar `do...while`

El caso idiomático es cuando la primera iteración es **incondicional**: siempre se debe ejecutar el bloque al menos una vez antes de verificar si continuar. Los casos más frecuentes son:

- Mostrar un menú y luego preguntar si el usuario quiere repetir.
- Pedir input al usuario y validarlo — siempre se pide al menos una vez.
- Inicializar un estado y luego comprobar si el resultado es válido.

En código moderno, `do...while` aparece con menos frecuencia que `while` porque muchos de sus casos se expresan con `while` precedido de una inicialización, o con funciones recursivas. Sin embargo, cuando la semántica de "ejecutar siempre la primera vez" es parte del diseño, `do...while` la hace explícita sin necesidad de flags.

## Recetas comunes

**Menú interactivo (Node.js con readline)**

```js
import * as readline from "readline/promises";

const rl = readline.createInterface({ input: process.stdin, output: process.stdout });
let continuar;

do {
  const opcion = await rl.question("Elige (1=Nuevo, 2=Abrir, 0=Salir): ");
  procesarOpcion(opcion);
  continuar = opcion !== "0";
} while (continuar);

rl.close();
```

**Validación con reintento — el prompt se muestra siempre primero**

```js
async function pedirEdad(rl) {
  let edad;
  do {
    const input = await rl.question("Edad (1-120): ");
    edad = parseInt(input, 10);
    if (isNaN(edad) || edad < 1 || edad > 120) {
      console.log("Valor no válido, intenta de nuevo.");
      edad = null;
    }
  } while (edad === null);
  return edad;
}
```

**Emular con `while` usando flag (menos idiomático)**

```js
// Equivalente menos claro — requiere flag artificial
let ejecutado = false;
while (!ejecutado || condicion) {
  ejecutado = true;
  // cuerpo
}
```

La versión con `do...while` expresa la intención directamente sin variables auxiliares.

## Diferencia con `while`

| Aspecto | `while` | `do...while` |
|---|---|---|
| Evaluación de condición | antes del cuerpo | después del cuerpo |
| Ejecuciones mínimas | 0 | 1 |
| Uso típico | cantidad desconocida, puede ser cero | primer paso obligatorio |

> [!tip] Buenas prácticas
> - Usar `do...while` solo cuando la semántica de "al menos una vez" forma parte del diseño — no como sustituto de `while` con una variable de estado extra.
> - No olvidar el punto y coma después de `while (condición);` — su ausencia no siempre produce un error inmediato pero puede causar comportamientos inesperados según el contexto.

> [!warning] Errores comunes
> **Punto y coma faltante** — `do { ... } while (cond)` sin `;` puede parsear correctamente en muchos contextos, pero si la siguiente línea comienza con `(`, el parser lo interpreta como una llamada a función:
> ```js
> do { algo() } while (cond)
> (function() { /* IIFE */ })()
> // Interpretado como: while(cond)(function(){})() → TypeError
> ```
> **Actualización de condición** — igual que en `while`, olvidar actualizar la variable de la condición dentro del cuerpo produce un bucle infinito.
